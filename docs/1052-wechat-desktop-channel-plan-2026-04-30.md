# 1052 微信桌面自动化通道接入方案

## 1. 目标

本次改造在现有 1052 OS 中新增一个与“扫码微信通道”并列、明确区分的“微信桌面自动化通道”。

该通道基于 Windows 桌面微信 UI 自动化能力，目标是让 Agent 可以：

- 持续监听指定微信会话，尤其是微信群聊。
- 在群聊中正确识别 `@机器人` 触发，而不是把发送者全部退化成“未解析成员”。
- 将群名、发送者名、是否群聊、是否被 `@` 等上下文自动拼接进模型上下文，而不是直接污染前端消息正文。
- 支持每个微信群单独的提示词追加配置。
- 支持每个微信群单独的权限开关。
- 支持微信群聊专属记忆，并在微信群场景下注入上下文。
- 在非微信群聊场景下，把“微信群聊记忆”抽象成工具挂载给 Agent，明确说明何时使用。
- 支持 Agent 主动调用“微信桌面自动化发送工具”，把消息发到微信，而不是只会回答“不能发”。
- 尽量采用静默持续监听方式，避免频繁抢焦点、抢鼠标。

## 2. 与现有能力的关系

### 2.1 保留现有扫码微信通道

现有 `backend/src/modules/channels/wechat/` 是扫码接入型微信通道，走的是 API 化收发模型。

本次不会把桌面自动化微信硬塞进这个模块内部继续混用，而是新增独立模块，前端也做独立入口，避免：

- 配置项混杂。
- 状态混杂。
- 发送目标混杂。
- 用户误以为两个通道能力和稳定性完全一样。

### 2.2 新增并列通道

暂定新增通道名：

- 后端模块：`wechat-desktop`
- 前端展示名：`微信桌面自动化`

前端社交通道页将至少区分为：

- 微信扫码通道
- 微信桌面自动化通道
- 飞书
- 企业微信

## 3. 数据与配置设计

### 3.1 存储目录

新通道数据统一放到：

- `data/channels/wechat-desktop/`

暂定文件：

- `config.json`
- `sessions.json`
- `groups.json`
- `events.jsonl`
- `memory.json`
- `worker-state.json`

### 3.2 群配置模型

每个微信群配置至少包含：

- `groupId`
- `groupName`
- `enabled`
- `mode`
- `promptAppend`
- `allowTools`
- `allowMemoryWrite`
- `allowAutoReply`
- `mentionOnly`
- `updatedAt`

其中：

- `enabled=false` 表示该群不接入 Agent。
- `mode=chat` 表示仅普通对话。
- `mode=full` 表示允许完整 Agent 能力。
- `promptAppend` 为该群专属补充提示词，拼接到系统上下文。
- `mentionOnly=true` 表示仅被 `@` 时触发。

### 3.3 会话/联系人模型

每个监听目标至少记录：

- `sessionId`
- `sessionName`
- `sessionType`：`direct` / `group`
- `lastMessageAt`
- `lastSenderName`
- `lastMessagePreview`
- `listening`

## 4. 运行架构

### 4.1 Node + Python Worker

由于桌面微信自动化库是 Python 生态，本次采用：

- Node/TypeScript 负责 API、前端状态、Agent 集成、记忆系统接入。
- Python Worker 负责桌面微信监听、消息解析、消息发送、群信息采集。

通信方式优先采用：

- Node 启动常驻 Python Worker。
- Node 与 Worker 之间使用 JSON Lines over stdio 或本地 HTTP。

首选 `stdio JSONL`，原因：

- 更容易做单实例管理。
- 不额外占用对外端口。
- 易于把 Python 能力封装成内部服务。

### 4.2 Worker 职责

Python Worker 负责：

- 初始化微信桌面自动化库。
- 枚举或绑定监听会话。
- 持续监听新消息。
- 解析消息类型、群名、发送者、是否 `@机器人`。
- 提供发送文本/文件能力。
- 返回群列表、会话列表、状态和错误信息。
- 尽量维持静默监听，不频繁切换窗口。

### 4.3 Node 职责

Node 服务负责：

- 管理 Worker 生命周期。
- 暴露 REST API 给前端。
- 将群聊消息映射到 Agent 运行时上下文。
- 写入聊天记录。
- 决定是否触发自动回复。
- 注入群聊提示词和群聊记忆。
- 暴露 Agent 工具给模型调用。

## 5. 群聊消息处理策略

### 5.1 入站标准化

Worker 产出的消息事件应标准化为：

- `channel = wechat-desktop`
- `sessionType = direct | group`
- `sessionName`
- `groupName`
- `senderName`
- `senderId`（若能解析）
- `mentionedBot`
- `text`
- `attachments`
- `raw`

### 5.2 群聊上下文注入

群聊触发模型时，不直接把冗长元信息塞进用户正文，而是把正文与上下文拆开：

- 用户正文只保留真正发给机器人的消息文本。
- 系统/运行时上下文增加：
  - 当前通道为微信群聊。
  - 当前群名。
  - 当前发送者。
  - 当前消息是否通过 `@机器人` 触发。
  - 当前群的专属提示词。
  - 当前群的能力开关。

这样可保证：

- 前端消息展示更干净。
- 模型仍拿到完整环境信息。

### 5.3 `@机器人` 解析

群聊中被 `@` 时：

- 先解析原始消息中的 `@机器人` 片段。
- 从展示给模型的文本中移除 `@机器人` 前缀。
- 保留发送者与群名在运行时上下文中。

目标效果：

- 前端看到的是干净文本。
- 模型知道是谁在什么群里 `@` 它。

## 6. 权限与模式控制

### 6.1 群级开关

每个群单独控制：

- 是否启用接入。
- 是否仅响应 `@`。
- 是否允许完整工具。
- 是否允许写入微信群记忆。
- 是否允许 Agent 自动回复。

### 6.2 模式边界

`chat` 模式：

- 只进行对话回复。
- 不允许高风险工具。
- 不主动调用外部发送能力之外的复杂执行链。

`full` 模式：

- 可使用完整 Agent 能力。
- 可调用微信发送工具。
- 可在群级策略允许时写入微信群记忆。

## 7. 微信群记忆设计

### 7.1 新增记忆域

在现有长期记忆体系外，新增“微信群聊记忆”域。

该记忆与全局 memory 分离，按群维度存储。

建议存储模型：

- `groupId`
- `groupName`
- `memoryId`
- `title`
- `content`
- `source`
- `createdAt`
- `updatedAt`
- `active`

### 7.2 自动记忆

当消息来源就是某个微信群时：

- Agent 运行后若形成应写入的群内长期信息，可直接写入该群记忆。
- 不走“必须再确认一次”的传统建议流。

前提：

- 仅在群配置 `allowMemoryWrite=true` 时执行。
- 明显敏感内容仍应受限制。

### 7.3 用户显式要求

如果用户在微信群里明确说“记住这个”：

- 默认写入当前微信群记忆。

如果用户明确说“写入全局记忆 / 其他记忆系统”：

- 按用户指定目标写入，不强行留在群记忆。

### 7.4 工具化

当当前对话不是微信群上下文时：

- 不自动把某个微信群记忆塞进普通上下文。
- 改为挂载工具给模型：
  - 查询某微信群记忆
  - 写入某微信群记忆
  - 列出可用微信群记忆

并在系统提示词中说明：

- 只有当任务与某个微信群上下文明确相关时，才调用此类工具。

## 8. Agent 工具设计

计划新增工具：

- `wechat_desktop_list_sessions`
- `wechat_desktop_send_message`
- `wechat_desktop_send_file`
- `wechat_group_memory_list`
- `wechat_group_memory_read`
- `wechat_group_memory_write`

工具说明会写得比当前更明确，至少包含：

- 适用场景
- 何时该用
- 何时不该用
- 必填参数
- 群聊/私聊差异
- 对权限开关的依赖

## 9. 系统提示词优化

本次会同步整理 Agent 系统提示词和工具描述，重点解决：

- 现有社交通道能力说明过于混杂。
- 工具边界不够清晰。
- 模型不知道什么时候该发微信、什么时候只该回复聊天。
- 群聊记忆和全局记忆优先级不清楚。

优化后会明确：

- 渠道上下文优先级。
- 群聊场景下上下文来源。
- 微信桌面自动化工具的用途与限制。
- 记忆写入优先级。
- 敏感信息的处理边界。

## 10. 前端改造

前端 `SocialChannels` 页面将新增独立板块，不和现有扫码微信混在一起。

新板块至少包含：

- Worker 运行状态
- 微信窗口连接状态
- 会话列表 / 群聊列表
- 群级开关
- 群专属提示词编辑
- 群级权限模式
- 测试发送
- 最近事件 / 错误状态

界面目标：

- 一眼区分“扫码微信”和“桌面自动化微信”
- 群配置集中可见
- 会话状态和监听状态可见

## 12. 风险与不确定项

### 12.1 UI 自动化稳定性

桌面微信能力本质上依赖 UI 自动化，风险包括：

- 微信版本更新导致控件结构变化。
- 某些群成员信息无法 100% 结构化解析。
- 监听过程中偶发焦点抢占。

### 12.2 静默监听能力

优先研究是否能做到：

- 基于子窗口内容变化的持续监听。
- 避免高频切换可见窗口。

若底层库无法完全静默，则退而求其次：

- 降低监听打扰频率。
- 降低抢焦点概率。
- 保持功能可用并明确记录限制。

### 12.3 群成员解析

若底层库无法稳定给出群成员唯一 ID，则先保底使用：

- 群名
- 发送者显示名
- 消息文本

保证不再退化成“未解析成员”，但唯一标识能力可能受限。
