## 2. V6 审计结论

### 2.1 已实现

- 单核渐进披露主链路已落地：
  - `agent.service` 已切到 `P0 -> upgrade -> 续跑`
  - `P0` 仅保留 `request_context_upgrade`
  - 已支持升级事件 SSE：`requested / applying / applied / aborted`
- `AgentSettings` 五个新开关已落地：
  - `progressiveDisclosureEnabled`
  - `providerCachingEnabled`
  - `checkpointEnabled`
  - `seedOnResumeEnabled`
  - `upgradeDebugEventsEnabled`
- `TokenUsage` 扩展已落地：
  - `cacheReadTokens`
  - `cacheWriteTokens`
  - `upgradeOverheadInputTokens`
  - `upgradeOverheadOutputTokens`
  - `upgradeOverheadTotalTokens`
- `TokenUsagePanel` 已展示缓存和升级开销
- `Checkpoint` 读写、注入、会话 ID 归一化已落地
- 老会话 `seed checkpoint` 已落地：
  - 有 `compactSummary` 时走快路径
  - 无 `compactSummary` 时会同步生成 seed
- 首批 pack 主能力已可用：
  - `base-read-pack`
  - `repo-pack`
  - `search-pack`
- `request_context_upgrade` 的两条核心约束已落地：
  - 单轮最多 2 次升级
  - 单次最多 2 个 pack
- `terminal_run_readonly` 已落地
- UAPIs 轻量目录摘要已落地
- 聊天流中止已透传到底层 LLM 请求

### 2.2 部分实现

- Provider caching：
  - 已有设置项、usage 归一化和 `agent.cache-policy.service.ts`
  - 但 `providerCachingEnabled` 目前没有真正接入运行时决策
  - 也没有把 Anthropic / DeepSeek / MiniMax 的 provider-specific cache breakpoint 真正打进请求
  - 结论：只有“统计口径”和“预留结构”，没有完成 V6 里要求的真正缓存落地
- Seed 慢路径模型策略：
  - 当前慢路径直接走 `settings.llm`
  - 还没有实现 V5/V6 里说的“优先便宜模型”
  - 也没有做 `3` 次失败上限
  - 结论：seed 可用，但还不够按方案收敛
- Pack 实施边界：
  - V6 计划首批只做 `base-read / repo / search`
  - 当前 pack 注册表里已经额外定义了 `memory / skill / plan / data / channel`
  - 这些 pack 没有像首批三包那样完成同级别的验收、路由提示和专项验证
  - 结论：代码里有超前预埋，和 V6 的“首批只做三包”不完全一致
- 运行时可观测性：
  - 前端可看到升级事件
  - usage 面板能看到缓存/升级开销
  - 但后端还没有专门把“是否 P0、挂了哪些 pack、升级次数、seed 状态”写成独立 runtime log
  - 结论：前端可观测性已到位，后端日志可观测性不足

### 2.3 未实现

## 3. 与 V6 的偏差说明

### 3.1 偏差一：缓存比计划弱

V6 里把 provider caching 列成核心第一阶段能力，但当前实现还停在“开关 + usage 归一化 + 结构预留”，这块不能算完成。

### 3.2 偏差二：seed 比计划更粗

V6 里要求老会话 seed 走“快路径优先、慢路径便宜模型、失败上限”，当前只做到了“快路径优先 + 同步慢路径”，还没做便宜模型策略和失败次数治理。

### 3.3 偏差三：pack 注册表超前

V6 计划首批只验证三包，但当前代码已经把更多 pack 名称和工具映射放进去了。它们不一定会立刻出问题，但会让“首批范围”变得不那么干净。

## 4. 当前建议

### 4.1 下一优先级最高

- 真正接入 provider caching 运行时策略
- 给 seed 慢路径补“便宜模型 + 重试上限”
- 把后端 runtime log 补上

### 4.2 其次

- 收紧 pack 暴露边界，只保留首批三包或把其余 pack 明确标成 disabled
- 把聊天页内联升级提示抽成独立组件
- 补一个 checkpoint 调试卡片
